## llvm编译器项目
### 面试准备
#### 前端部分
1. 词法分析 (Lexical) 与语法分析 (Syntax)
流程：parser/lexer.c 和语法解析规则（Flex/Bison 生成）。将文本流转换为 Token 流，然后依靠语法规则生成抽象语法树（AST）。
数据结构：在你的项目里，这对应 common/syntax_tree.h 和 common/ast.hpp。ast.hpp 定义了各种树节点，比如 BinaryExpr（加减乘除）、IfStmt（条件分支）、WhileStmt（循环）等。
1. AST 遍历与环境状态维护 (Visitor Pattern + Scope)
生成 IR 的核心机制是访问者模式（Visitor Pattern）。

在 sysy_builder.hpp 中，应该有一个构建器（Builder）类继承自 AST 的 Visitor。它重载了 visit(IfStmt* )、visit(BinaryExpr* ) 等方法。
它需要维护一个符号表（Symbol Table），记录当前作用域内的变量名到其分配的内存地址或 Value* 的映射。这样遇到 a = b + 1 里面的 b 时，能知道去哪个地址 Load 数据。
3. IRBuilder 与指令的生成（最关键的一步）
项目目录中有一个极其重要的文件：IRBuilder.hpp。它的作用就像是一个在流水线上的“打字机”。

维护插入点（Insert Point）：IRBuilder 内部保存了当前正在操作的 BasicBlock（基本块）。当你调用 builder->create_add(v1, v2) 时，它会自动生成一条对应的加法 Instruction，并将其 push_back 到当前基本块的指令链表（ilist）末尾。
4. 具体控制流结构如何生成？ (If / While)
这是面试常考的重点！如果在 AST 层遇到一个 while(exp) { body }，你的 Builder 会怎么生成 LightIR 的图？

生成新的基本块：用 BasicBlock::create(...) 创建至少 3 个基本块：cond_block（条件块）、body_block（循环体块）、next_block（退出循环的块）。
连接跳转指令（Branch）：
在进入 while 之前，生成一条无条件跳转指令跳转到 cond_block。
让 IRBuilder 的插入点移动到 cond_block，遍历生成 exp （条件表达式）的 IR。
根据 exp 的结果（Value*），利用 IRBuilder 生成一条条件跳转指令（Br cond_val, body_block, next_block）。
处理循环体：
将插入点移动到 body_block，递归调用 Visitor 处理循环体内部的 AST 节点。
循环体末尾，插入无条件跳转，跳回 cond_block。
结束点：将插入点移动到 next_block，继续处理后续的 AST。

“面试官您好，在这个 segfault-compiler 项目中，我参与实现了一个完整的、类 LLVM 架构的 SysY 语言编译器，能够将类 C 语言编译为 RISC-V/LoongArch 机器码。底层基于我们自己设计的 SSA 形式中间表示 LightIR。
在项目中，我端到端地打通了多项核心优化，主要负责了中端的过程间优化（函数内联）、循环优化（循环不变式外提 LICM）以及后端的线性扫描寄存器分配（Linear Scan）。我的目标是权衡编译耗时与最终生成的指令运行效率。”

【冲突与挑战：遇到了什么 Bug/瓶颈？怎么解决的？】

“在开发过程中，我遇到的**最大性能瓶颈和挑战来自于『函数内联与寄存器分配的连锁反应（相克）』**。

比如在起初做函数内联（Function Inlining）时，我采取了比较激进的策略。虽然这消除了函数调用的栈开销，但由于代码急剧膨胀（Code Bloat），导致了一个致命问题：变量的活跃区间（Live Interval）被拉得非常长，这使得同时活跃的变量数远远超过了物理寄存器的数量上限。
这直接压垮了我写的线性扫描寄存器分配器，导致后端生硬地把大量变量 Spill（溢出）到内存中。最终的测试结果是：激进内联后的代码，运行速度反而比没优化前还要慢，因为大量的访存指令吃掉了原本函数调用省下的时间。

【方案与结果：如何解决？】

为了解决这个问题，我做了两个改进：

在内联侧引入 Cost Model（开销成本模型）：我设计并实现了一套基于启发式规则的函数内联 Cost Model。该模型通过在编译前端动态测算目标函数（Callee）的指令规模作为惩罚，如果参数中有常量则作为奖励（可以引发常量传播优化），并设置内联深度限制，来评估内联的潜在收益与风险。只有当内联带来的性能提升明显超过代码膨胀引起的寄存器压力时，才会执行内联。

与此同时，我的函数内联把原本隐藏在函数调用内的代码平铺开后，极大地增加了**LICM（循环不变式外提）**的优化机会。内联暴露出的常数和循环内不变的地址计算，被 LICM 成功提到了循环前（Preheader）执行。最终这三个 Pass 形成了一个闭环，在编译时间和动态执行指令数上取得了很好的平衡，大幅提高了程序的性能水平。”

## mlir项目（fhe编译优化实现）
### 自举算子插入优化
基于动态规划算法优化自举算子的排布，性能相比传统手动插入提升20%。
- dp算法：这个 Pass 是我在基于 MLIR 构建的同态加密（HE）编译器中，为解决自举（Bootstrapping）操作最优插入问题而设计的核心优化模块。在同态加密计算中，自举操作耗时极高，同时长程数据依赖（Shortcuts）为了满足密文层级（Level）对齐的硬性要求，会强行压低沿途其余计算节点的可用层级，导致整体计算延迟剧增。为了最小化程序的端到端执行时间，我将该节点选址问题抽象为动态规划（DP）模型：用 DP 数组记录沿着计算图深度推进时，在每一层发生自举的最小累计延迟。在设计状态转移方程时，为了解决长程边带来的层级挤压矛盾，我引入了长程边子集枚举与代价评估策略，在‘保留长程边但牺牲同深度普通节点计算速度’与‘强制打断长程边增加单次自举开销以换取局部计算提速’之间进行精确的 Trade-off。最终，算法通过回溯 DP 路径，全局寻找到了自举开销与密集计算延迟之和最小的最优插入边界点与必须被阻断的长程边集合，并在 IR 层级为对应节点打上 Boot=true 标签，为后端代码生成提供了极具性能优势的自举调度策略。
> 算法详细拆解：
> 1. 明确 DP 状态定义与转移思路
“首先，我设定 DP 数组 dp[i] 来表示：当计算图推进到深度 i 并且恰好在第 i 层执行了一次自举操作时，整个程序端到端的最小累计延迟。
在计算 dp[i] 时，我们需要回头去寻找上一个最优的自举位置 j。那么状态转移方程的核心，就是如何精确计算在区间 [j, i] 之间不发生自举的期间，所产生的区间总延迟代价。”
>2. 解释引入难点：长程边（Shortcuts）与层级挤压矛盾
“计算区间代价时，最大的阻碍来自于跨越该区间的长程数据依赖（Shortcuts）。在同态加密的机制中，密文的每次乘法都会消耗可用的层级（Level），层级耗尽就必须做自举。
如果有一条长程边跨越了 [j, i] 这一长段区间，为了让这个数据能活着送达底层而不报错，在层级对齐机制下，它会强迫压低沿途经过的所有深度的可用层级。而同态加密中，节点在越低的层级下执行，耗时就越长。这就是所谓的‘层级挤压’导致普通节点变慢的问题。”
>3. 详细拆解 Trade-off（权衡决策）
“为了打破这个僵局，我们在处理每一条长程边时，其实都面临一个极端的 Trade-off：”
策略 A（保住长程边）：选择顺应长程边，不在起点 j 把它干掉。代价是：它会触发层级重新分配，拖慢区间内所有普通节点的计算速度。
策略 B（强制打断长程边）：选择在起点 j 强制让这条长程边也执行一次自举运算。代价是：承受一次巨大的固定自举延迟开销；收益是：消除了它对局部层级的束缚，区间内的普通节点可以分配到更高的层级，从而获得计算上的大幅提速。”
>4. 算法落地：子集枚举与精确代价评估
“由于现实计算图中，跨越某个区间的长程边往往不止一条，这是一个典型的组合优化子问题。因此我设计了长程边子集枚举机制：
对于当前评估的区间 [j, i]，我会先收集所有穿过它的长程边，然后枚举它们所有可能的保留组合（子集）。针对每一个子集，我都会进行一次模拟评估：
计算代价 = 『未能保留的长程边需要额外执行自举的开销总和』 + 『剩余节点在新的受限层级下的实际计算耗时总和』。
通过比较所有子集的结果，算法能动态地判断出：在当前的图结构和层深下，到底放弃哪些长程边、保留哪些长程边才是全局最划算的。最后取出这个最优子集的代价，作为 DP 状态转移的最优成本，从而彻底解决层级分配与开销最小化之间的矛盾。”

## cuda编程项目
针对 SNN 多时间步迭代导致访存密集的问题，独立设计并实现了一个基于 LeNet 架构的 CUDA SNN 前向推理引擎，支持 FashionMNIST 数据集的图像分类
### SNN介绍
与CNN不同，需要在时间维度上进行迭代，导致访存密集，性能较差，比如项目中需要循环8次

### GPU架构
GPU的核心是SM（Streaming Multiprocessor），每个SM包含多个CUDA核心，负责执行并行计算任务。
1. 逻辑结构：由grid，block和thread组成，线程是GPU执行的基本单位，线程块是线程的集合，网格是线程块的集合。
2. 内存层次：包括寄存器、共享内存、全局内存等，不同类型的内存具有不同的访问速度和范围。
3. 物理结构：GPU由多个SM组成，每个SM包含多个warp，每个warp中的（32个）线程同时执行相同的指令。
4. bank冲突：shared memory被划分为32个bank， 如果在$32 \times 32$ 的共享内存矩阵中，线程访问的地址在同一个bank中，就会发生bank冲突，导致性能下降。使用padding可以避免bank冲突。

### 项目细节
项目描述：使用 CUDA C++ 从头实现并优化了一个 SNN (脉冲神经网络) 的前向推理引擎，对 FashionMNIST 数据集进行分类。
网络架构：LeNet-5 变种 (Conv1 -> Pool1 -> Conv2 -> Pool2 -> FC1 -> FC2 -> FC3)。时间步长 T=8
核心优化点：
算子融合 (Kernel Fusion) / 持久化核函数 (Persistent Kernel)：没有调用 cuDNN 等库，而是直接手写了一个巨大的 persistent_infer_kernel，将所有的卷积、池化、全连接层以及 SNN 的时间循环 (T=8) 全部融合在一个 Kernel 中。避免了频繁的 Kernel launch 开销和中间变量在全局内存 (Global Memory) 中的反复读写。
细粒度并行映射：采用 Batch 并行策略，1 个 Block 处理 1 个样本 (blockIdx.x + batchOffset)。Block 内的 256 个 Threads 协同处理特征图和全连接层向量。
共享内存 (Shared Memory) 优化：使用 Shared Memory 协同加载整个输入图像 (28x28) 以及中间的 Pool1 结果。因为图像和特征图在卷积核滑动时会被重复读取，放进 Shared memory 极大降低了 Global Memory 带宽压力。
寄存器缓存与循环展开 (Register Caching & Unrolling)：在算子计算时，将小尺寸的卷积核窗口 (如 5x5 的 Conv1_K) 先手动提取入片上寄存器 (Register Array patch[CONV1_K * CONV1_K]) 再进行计算，配合 #pragma unroll 提升指令级并行度 (ILP)。
内联 PTX 汇编优化：通过手动编写内联汇编 ptx_fma_f32 强制发射 fma.rn.f32 (乘加融合指令)，最大化 ALU 计算吞吐量。
🤔 面试高频问题模拟 (Q&A)
1. 为什么采用一个大 Kernel (Persistent Kernel) 把所有层写在一起，而不是逐层 Launch Kernel？
回答要点：

消除 Launch Overhead：每次调用 Kernel 都有微秒级的开销。特别是在 SNN 中，计算包含一个时间循环 (T=8)，如果每层、每个时间步都 launch，相当于要几百次 launch，开销占比巨大。
数据局部性 (Data Locality)：融合在一块后，前一层的结果（或者说特征图甚至膜电位）可能还在 L1/L2 Cache 或者 Shared Memory 中，下一层直接读取处理，显著降低了昂贵的 Global Memory 访存开销。
1. 在这个 CUDA 程序里，你是如何使用 Shared Memory 优化的？
回答要点：

用途：主要用于缓存输入图像 (s_input) 和池化层1的输出 (s_pool1)。
原因：卷积操作具有极强的数据重用性 (Spatial Locality)。相邻的输出像素需要读取大面积重叠的输入像素。如果直接从 Global Memory 读，带宽会成为瓶颈。
实现：Block 启动后，首先让 256 个线程通过步进 (for (int i = tid; i < size; i += nthreads)) 协同搬运 Global Memory 的数据到 Shared Memory 中，然后调用 __syncthreads() 同步，最后各个计算线程都从 Shared memory 去取 patch 数据。
3. 你的 Block / Thread 并行划分策略是什么？有哪些利弊？
回答要点：

策略：采用 grid(bs), block(256)。即一个 Thread Block 负责计算 Batch 中的一张完整的图片，Block 内的 256 个 Thread 根据元素的全局索引 (idx = tid; idx < total; idx += nthreads) 分摊该层各个特征图/神经元的计算任务。
利 (优点)：易于共享该图片的中间数据（通过 Shared Memory 存中间特征图）。无需不同 Block 之间的同步，因为每个样本的推断是独立的。
弊 (潜在改进点)：256 个线程去平摊并行度，对于全连接层或较小特征图的情况可能导致资源利用不足。更大规模的网络可能需要 "把特征图拆分给多个 Block" 或者使用 Tensor Core。
4. 代码中看到了 ptx_fma_f32，为什么要手写 PTX 汇编指令？
回答要点：

FMA 表示 Fused Multiply-Add (d=a×b+c)。
在硬件底层，NVIDIA GPU 对乘加操作有专门的 FMA 指令执行单元。如果不加干预，编译器有时为了严格遵循浮点标准可能不会自动将乘法和加法折叠。
通过手动使用内联汇编强制发射 fma.rn.f32 指令，可以确保这两个运算在一个时钟周期内完成（并且只舍入一次提高精度），从而把计算逻辑的吞吐量压榨到极限。该手段常用于卷积、矩阵乘这类典型的计算密集型算子。
1. SNN (脉冲神经网络) 的计算和传统 CNN 在这段代码里的体现有什么区别？
回答要点：

网络增加了时间维度 for (int t = 0; t < T; ++t)。
增加了在时间维度上持久存在的膜电位 (Membrane Potential) 数组（如 d_conv1_pot）。
激活函数由 ReLU 变成了 LIF (Leaky Integrate-and-Fire) 模型的核心逻辑：即电压 V 累加输入后，判断 if (V > threshold)，如果超过阈值发出脉冲 (返回 1.0f 或者 boolean)，并将电压重置。脉冲(spike)会被作为下一层的输入传递。
